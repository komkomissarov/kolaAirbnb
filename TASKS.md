# Задачи проекта LuxRent — декомпозиция

Кратко: сайт бронирования жилья. Реализованы регистрация, авторизация и логин. Нужно довести до рабочей версии: админка (пользователи), фронт с учётом авторизации, каталог из БД и заявки на бронирование.

Структура БД — см. `db.sql`: таблицы `users`, `properties`, `bookings`.

---

## 1. Фронт: отображение состояния авторизации

**Проблема:** После входа на страницах по-прежнему показывается «Войти», у админа нет перехода в админ-панель.

### 1.1 API проверки сессии

- **Задача:** Добавить endpoint, который по текущей сессии возвращает данные пользователя для фронта.
- **Действия:**
  - Создать `backend/api/auth-check.php` (или `api/me.php`): `session_start()`, проверить `$_SESSION['user_id']`. Вернуть JSON: `{ "loggedIn": true/false, "role": "user"|"admin"|null, "name": "..." }`.
  - Вызывать только GET (или POST без изменения данных).
- **Критерий готовности:** Запрос с активной сессией возвращает `loggedIn: true`, без сессии — `loggedIn: false`.

### 1.2 Единый блок «Войти / Профиль / Админ» в шапке

- **Задача:** На всех публичных страницах показывать в шапке разное в зависимости от авторизации.
- **Страницы:** `index.html`, `catalog.html`, `reg.html`.
- **Логика:**
  - При загрузке страницы вызвать `backend/api/auth-check.php` (с `credentials: 'include'` если нужны куки).
  - Если не авторизован: кнопка «Войти» (ссылка на `reg.html`).
  - Если авторизован обычный пользователь: показать «Профиль» (или имя) и «Выйти» (ссылка на `backend/admin/logout.php` или отдельный `logout.php` в корне).
  - Если роль `admin`: дополнительно кнопка/ссылка «Админ-панель» → `backend/admin/dashboard.php`.
- **Реализация:** Один общий JS-файл (например `auth-header.js`) или встроенный скрипт в каждой странице: в блок `.auth-buttons` подставлять разметку в зависимости от ответа API.
- **Критерий готовности:** После входа на главной и в каталоге видно «Выйти» и (для админа) «Админ-панель»; кнопка «Войти» скрыта.

### 1.3 Выход для обычного пользователя

- **Задача:** Сделать выход доступным не только из админки.
- **Действия:** Либо использовать существующий `backend/admin/logout.php` (он только убивает сессию и редиректит — можно редиректить на `index.html` для не-админов), либо добавить в корень `logout.php`, который делает `session_destroy()` и редирект на главную. На фронте ссылка «Выйти» ведёт на этот скрипт.
- **Критерий готовности:** После нажатия «Выйти» пользователь разлогинен и видит снова «Войти».

## 2. Админ-панель: отображение пользователей

**Проблема:** В админке только список недвижимости, нужен список пользователей.

### 2.1 Страница списка пользователей

- **Задача:** Вывести в админке список пользователей из таблицы `users`.
- **Действия:**
  - Создать `backend/admin/users.php`: доступ только для `role === 'admin'`, подключить `db.php`, `SELECT id, name, email, role, created_at FROM users ORDER BY created_at DESC`, вывести в виде таблицы или карточек. Пароли не выводить.
  - В `backend/admin/dashboard.php` добавить ссылку «Пользователи» на `users.php`.
- **Критерий готовности:** В админке есть пункт «Пользователи», по клику открывается страница со списком пользователей (имя, email, роль, дата).

### 2.2 (Опционально) Навигация админки

- **Задача:** Явное меню: «Недвижимость» (текущий dashboard) и «Пользователи».
- **Действия:** В `dashboard.php` и `users.php` добавить общий блок навигации (табы или ссылки) между разделами.
- **Критерий готовности:** Из раздела недвижимости можно перейти в пользователи и обратно.

## 3. Каталог: данные из БД и заявки на бронирование

**Проблема:** В `catalog.html` используется локальный массив-заглушка; нужно брать данные из БД и дать возможность оставлять заявки.

### 3.1 Загрузка каталога из API

- **Задача:** Убрать из `catalog.html` массив `properties`, подгружать список из бэкенда.
- **Действия:**
  - Оставить/использовать существующий `backend/api/properties.php` (уже отдаёт JSON из `properties`).
  - В `catalog.html` при загрузке страницы: `fetch('backend/api/properties.php')`, затем отрисовать карточки из ответа.
  - Привести формат карточек к полям БД: `id`, `title`, `city`, `price`, `description`, `image`. В текущей БД нет полей `type`, `rooms`, `location`, `features` — либо отображать только имеющиеся поля, либо позже расширить схему (отдельная задача).
  - Обработать пустой ответ и ошибку сети (сообщение «Нет объектов» / «Ошибка загрузки»).
- **Критерий готовности:** Карточки в каталоге формируются из ответа API, без захардкоженного массива.

### 3.2 Фильтр по городу по данным из БД

- **Задача:** Фильтрация по городу должна работать с данными из API.
- **Действия:**
  - Вариант А: загружать все объекты один раз и фильтровать на фронте по полю `city`.
  - Вариант Б: добавить в `backend/api/properties.php` опциональный параметр `?city=Москва` и фильтровать в SQL; фронт при смене фильтра перезапрашивает API.
- **Критерий готовности:** Выбор города в фильтре ограничивает список объектов (из БД).

### 3.3 Форма заявки на бронирование

- **Задача:** Пользователь может оставить заявку на бронирование выбранного объекта.
- **Действия:**
  - На каждой карточке (или в модальном окне по клику) добавить кнопку «Оставить заявку» / «Забронировать».
  - Форма: имя, телефон, `property_id` (подставлять из карточки). Отправка POST на `backend/api/booking.php` с телом JSON `{ "name", "phone", "property_id" }` (как уже реализовано в `booking.php`).
  - В `backend/api/booking.php`: добавить заголовок `Content-Type: application/json`, проверку входных данных, обработку ошибок (например, отсутствие поля), ответ с полем `message` при успехе/ошибке.
  - После успешной отправки показать сообщение «Заявка отправлена» и закрыть форму/модалку.
- **Критерий готовности:** С карточки объекта можно отправить заявку, данные попадают в таблицу `bookings`, пользователь видит подтверждение.

## 4. Исправления и доработки

### 4.1 Подключение БД в админке

- **Задача:** Исправить путь к `db.php` в файлах админки.
- **Действия:** В `backend/admin/dashboard.php`, `backend/admin/add-property.php`, `backend/admin/delete-property.php` заменить `require_once __DIR__ . '../../../db.php'` на `require_once __DIR__ . '/../../db.php'` (чтобы корректно подниматься на два уровня вверх от `backend/admin/`).
- **Критерий готовности:** Админка без ошибок подключает БД при запуске из корня проекта.

### 4.2 Пароль админа в БД (безопасность)

- **Задача:** В `db.sql` пароль админа задан открытым текстом (`'admin123'`). Для продакшена нужен хэш.
- **Действия:** Сгенерировать хэш (например `password_hash('admin123', PASSWORD_DEFAULT)` в PHP), заменить в `db.sql` в INSERT для users. Либо оставить отдельную инструкцию в README для первого входа (сменить пароль после деплоя).
- **Критерий готовности:** В схеме/инструкции пароль админа не хранится в открытом виде или есть явное предупреждение в README.
